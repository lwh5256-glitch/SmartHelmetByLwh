# DS18B20 与 GPS 接入思路

## 1. 总体思路

新增模块时不要只看 MCU 有没有空闲引脚，还要同时看三件事：

1. 这个引脚是否已经被当前 Keil 工程中的代码占用
2. 探索者 V3 板上这个引脚是否还挂着别的板载资源
3. 新模块是否必须独占串口/总线

本项目按这个原则推下来，最合理的结果是：

- DS18B20 走 `PG9` 单总线
- GPS 走新的独立串口 `USART6(PC6/PC7)`

## 2. 为什么 DS18B20 不用两个 GPIO

DS18B20 是标准单总线器件，本来就支持多点挂接。

所以两个 DS18B20 的正确接入思路不是：

- 传感器 1 接一个 GPIO
- 传感器 2 再接一个 GPIO

而是：

- 两个探头共用一个 `DQ`
- 软件通过 ROM 编号识别不同探头

这样做的好处：

- 节省 IO
- 接线简单
- 扩展到 3 个、4 个探头时不需要继续改硬件

## 3. 为什么 DS18B20 选 PG9

因为探索者 V3 已经把 `PG9` 做成了板上的单总线接口：

- 明确支持 `DHT11/DS18B20`
- 已带 4.7K 上拉

而你当前工程里：

- `DHT11` 驱动虽然编译进去了
- 但主程序根本没有真正使用它

所以直接把 `PG9` 资源切换给 DS18B20，是最自然的方案。

## 4. DS18B20 的接入步骤建议

### 第一步

先只接 1 个 DS18B20，验证：

- 复位脉冲正常
- 能读到 ROM
- 温度值正常变化

### 第二步

再把第 2 个 DS18B20 并到同一根 `PG9` 总线上，验证：

- 能搜索到两个不同 ROM
- 读数不会串

### 第三步

把 ROM 和逻辑身份绑定，例如：

- ROM_A -> 体表温度
- ROM_B -> 环境温度

不要依赖“搜索顺序”来区分两个探头。

## 5. 为什么 GPS 不能继续走 USART3

ATK-MO1218 官方探索者 F407 示例默认使用 `USART3(PB10/PB11)`，这是因为它默认插在 ATK-MODULE 接口上。

但你当前项目已经把：

- `PB10/PB11`
- `USART3`

全部给了 ESP8266。

如果硬接进去，会出现：

- 串口冲突
- 中断冲突
- ESP8266 和 GPS 数据混流

所以 GPS 必须换到新串口。

## 6. 为什么推荐 GPS 用 USART6

相比其他可选串口，`USART6(PC6/PC7)` 更适合当前项目：

- 不占用当前已经工作的 PMS7003 串口
- 不占用当前已经工作的 ESP8266 串口
- 不碰 PG9 单总线口
- 不依赖板上的 CH340 调试串口

这意味着你只需要：

- 新增一个 UART 初始化
- 新增一个 GPS 接收缓冲
- 新增一个 NMEA 解析流程

不用重构现有 ESP8266 / PMS7003 驱动。

## 7. GPS 的接入步骤建议

### 第一步

先只做串口收数：

- 配好 `USART6`
- 串口中断收一整帧
- 确认能收到 `$GNRMC`、`$GNGGA` 之类的 NMEA 语句

### 第二步

只解析最关键的几个字段：

- 定位是否有效
- 纬度
- 经度
- UTC 时间
- 卫星数

### 第三步

再把数据显示到 LCD，并决定是否上传云端。

不建议一开始就把：

- 串口改造
- NMEA 解析
- LCD 页面改版
- 云端数据格式改造

全部同时做掉，不然调试定位很难。

## 8. 软件结构建议

### DS18B20

- 新建 `ds18b20.c/.h`
- 提供总线复位、读写位、读写字节、搜索 ROM、启动转换、读取温度
- 对外暴露“按 ROM 读取两个探头温度”的接口

### GPS

- 新建 `gps_uart.c/.h` 或 `atk_mo1218_port.c/.h`
- 专门管理 `USART6`
- 再新建 `gps_parse.c/.h`
- 将“串口收包”和“协议解析”拆开

## 9. 建议的实施顺序

1. 先接 DS18B20，两只共用 `PG9`
2. 软件先完成单总线搜索与双探头读温
3. 再新增 `USART6`
4. 接入 GPS 并完成 NMEA 收包
5. 最后统一修改 LCD 显示和数据上报

这个顺序风险最低，也最容易逐步验证。
