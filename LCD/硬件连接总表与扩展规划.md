# 硬件连接总表与扩展规划

## 1. 判定依据

本表不是只按仓库目录判断，而是按以下三类信息交叉确认：

1. `MDK-ARM/LCD.uvprojx` 中实际参与编译的文件
2. `Core/Src/*.c`、`Drivers/HARDWARE/*`、`MyCode/*` 中真实生效的引脚定义
3. `资料/探索者V3 IO引脚分配表.xlsx` 中探索者 V3 开发板板级资源分配

## 2. 先说结论

### 2.1 当前工程真实生效的外设

- LCD：FSMC 16bit 总线
- MPU6050：软件 I2C
- PMS7003：USART1
- ESP8266：USART3
- 蜂鸣器：PF8
- 按键：PA0、PE2、PE3、PE4
- LED1/LED2：PF9、PF10

### 2.2 与 `项目总结.md` 不一致的地方

- `项目总结.md` 写 MPU6050 在 `PB8/PB9`，但实际工程代码使用的是 `PB6/PB7`
- `DHT11` 驱动已加入 Keil 工程，DQ 定义在 `PG9`，但 `main.c` 中并未调用 `dht11_init()` / `dht11_read_data()`，因此目前没有真正启用 DHT11

## 3. 当前工程真实引脚占用

| KEY_UP | PA0 | 高电平按下 |

| ESP8266 | PB10 / PB11 | USART3_TX / USART3_RX |
| LCD 背光 | PB15 | `LCD_BL` |
| KEY2 | PE2 | 低电平按下 |
| KEY1 | PE3 | 低电平按下 |
| KEY0 | PE4 | 低电平按下 |
| 蜂鸣器 | PF8 | 推挽输出 |
| LED1 | PF9 | 状态灯 |
| LED2 | PF10 | 心跳灯 |

### 3.2 LCD 的 FSMC 占用

| 功能 | 引脚 |
|---|---|
| FSMC_D0~D3 | PD14 / PD15 / PD0 / PD1 |
| FSMC_NOE / NWE | PD4 / PD5 |

说明：

- LCD 还占用了 `PB15` 作为背光控制
- 因此 `PD0/1/4/5/8/9/10/14/15`、`PE7~PE15`、`PF12`、`PG12` 不再适合分给新增模块

### 3.3 已编译但当前未真正使用的单总线资源

| 模块/资源 | 引脚 | 当前状态 |
|---|---|---|
| DHT11 驱动 | PG9 | 已编译，未在 `main.c` 中调用 |
| 探索者板载单总线接口 U19 | PG9 | 板上已给单总线接口预留，并带 4.7K 上拉 |

## 4. 板级资源对新增模块的影响

### 4.1 DS18B20

探索者 V3 的 `PG9` 本身就是板上的单总线接口，资源表明确说明：

- `PG9` 对应 `1WIRE_DQ`
- 同时兼容 `DHT11/DS18B20`
- 板上已经带 `4.7K` 上拉

这意味着：

- 两个 DS18B20 最合适的接法不是占两个 GPIO
- 而是共用一个单总线口 `PG9`
- 通过每个器件独有的 64 位 ROM 编号区分 1 号和 2 号温度探头

### 4.2 GPS

正点原子 ATK-MO1218 在探索者 F407 官方示例中默认使用 `USART3(PB10/PB11)`，这是因为官方默认让 GPS 模块插 ATK-MODULE 接口。

但你当前工程中：

- `USART3(PB10/PB11)` 已经被 ESP8266 占用
- 因此 GPS 不能继续按官方默认接到 ATK-MODULE 口

必须为 GPS 单独腾出另一组串口。

## 5. 新增模块推荐接线方案

## 5.1 两个 DS18B20 推荐方案

### 方案结论

两个 DS18B20 共用 `PG9` 一根数据线。

### 接线表

| DS18B20 引脚 | 连接到 |
|---|---|
| VDD | 3.3V |
| GND | GND |
| DQ | PG9 |

两只都这样接，DQ 直接并联到同一根总线上。

### 说明

- 推荐使用外部供电模式，不建议寄生供电模式
- 如果你直接接探索者板的单总线接口 U19，通常不需要额外再补 4.7K 上拉
- 如果你飞线到别的位置，仍然建议保留 `DQ -> 3.3V` 的 4.7K 上拉
- 软件上通过搜索 ROM 编号，把两个探头固定命名为 `DS18B20_1`、`DS18B20_2`

## 5.2 GPS 推荐方案

### 推荐主方案

把 GPS 单独迁移到 `USART6`：

- `PC6 -> GPS_RXD`
- `PC7 -> GPS_TXD`

### 接线表

| GPS 模块引脚 | 连接到 |
|---|---|
| VCC | 5V 或按模块手册要求供电 |
| GND | GND |
| TXD | PC7（MCU RX） |
| RXD | PC6（MCU TX） |

### 为什么优先推荐 USART6

- 不和当前 `PMS7003(USART1)` 冲突
- 不和当前 `ESP8266(USART3)` 冲突
- 不占用 `PG9` 单总线口
- 比改用 `USART2(PA2/PA3)` 更干净，因为 `PA2/PA3` 在板级还牵涉到 RS232/RS485 与以太网复用

### 对板级资源的要求

`PC6/PC7` 在探索者 V3 上与音频/摄像头复用，但你当前工程没有使用这些资源，因此可作为 GPS 专用串口。

### 不建议方案

- 不建议继续使用 `PB10/PB11`
  原因：该口已经给 ESP8266 用了
- 不建议把 GPS 塞到 `PA9/PA10`
  原因：该口当前已经给 PMS7003 使用
- 不建议给两个 DS18B20 分两个独立 GPIO
  原因：单总线本来就支持多点挂接，没必要额外浪费 IO

## 6. 整机连接汇总

## 6.1 扩展后推荐连接总表

| 模块 | 接口/引脚 | 备注 |
|---|---|---|
| LCD | FSMC + PB15背光 | 现状保持不变 |
| MPU6050 | PB6 / PB7 | 现状保持不变 |
| PMS7003 | USART1: PA9 / PA10 | 现状保持不变 |
| ESP8266 | USART3: PB10 / PB11 | 现状保持不变 |
| DS18B20 #1 | PG9 单总线 | 与 #2 共线 |
| DS18B20 #2 | PG9 单总线 | 与 #1 共线 |
| GPS | USART6: PC6 / PC7 | 新增 |
| 蜂鸣器 | PF8 | 现状保持不变 |
| KEY_UP | PA0 | 现状保持不变 |
| KEY0 | PE4 | 现状保持不变 |
| KEY1 | PE3 | 现状保持不变 |
| KEY2 | PE2 | 现状保持不变 |
| LED1 / LED2 | PF9 / PF10 | 现状保持不变 |

## 6.2 供电建议

| 模块 | 推荐供电 |
|---|---|
| DS18B20 x2 | 3.3V |
| GPS 模块 | 按模块板卡要求，优先看模块板载稳压范围 |
| ESP8266 | 保持原方案 |
| PMS7003 | 保持原方案 |
| MPU6050 | 保持原方案 |

说明：

- DS18B20 直接接 3.3V 最稳妥
- GPS 若是带底板的模块，常见可以接 5V 供电、串口逻辑为 3.3V TTL
- 真正下手接线前，仍建议你再用模块丝印或手册封面页确认一下 GPS 板的 `VCC` 输入范围

## 7. 建议的软件改动方向

### 7.1 DS18B20

- 不再启用 DHT11 采集流程
- 直接复用 `PG9` 单总线资源
- 新建 `Drivers/HARDWARE/DS18B20`
- 初始化时先搜索总线上的两个 ROM
- 按 ROM 固定映射为“探头1 / 探头2”
- 周期读取温度并显示/上报

### 7.2 GPS

- 新增 `USART6` 初始化与中断
- GPS 建议先只做 NMEA 接收解析
- 至少解析 `RMC` / `GGA`
- LCD 上可增加：定位状态、经纬度、卫星数、UTC 时间
- 云端上报建议后面再接，不要和底层接线改动混在一起

## 8. 最终建议

本项目最稳妥的扩展方式是：

- 两个 DS18B20 共用 `PG9` 单总线
- GPS 单独走 `USART6(PC6/PC7)`
- 保持 `PMS7003` 与 `ESP8266` 原接线完全不动

这样改动最小，硬件冲突最少，后续软件结构也最清晰。
